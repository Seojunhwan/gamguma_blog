## why

사용하기 어려운 애자일 소프트웨어 개발론을 툴을 사용함으로써 쉽게 도입할 수 있도록 해준다.

칸반보드를 통해 진행 상황을 쉽게 확인할 수 있음

개발자 도구 통합

## 애자일

> 짧게 짧게 진행하는 스프린트? 변화에 빠르게 대응?

### 스크럼?

계획하고 하는 것이 아닌 짧게 짧게 계획, 개발을 통해 변화에 대응이 가능

스크럼 기반으로 프로젝트 진행
데일리 스크럼(지금의 데일리 미팅) 을 통해 태스크를 정리

스크럼 마스터의 역할이 중요

1~2주 단위로 프로젝트 진행 후 회고
백로그에서 이번 스프린트에 넣을 태스크들을 넣고 1~2주 스프린트 시작
스프린트 기간 동안엔 팀의 화합과 협업을 중시하며 문제해결에 집중
스프린트 회고에 멘토가 참여하여 코칭 진행

현재 저희는 1주 단위로 스프린트 진행 중

### 1단계

- 애자일 소프트웨어 개발
- 스크럼 스프린트 kpt 회고
- 지라 등등
- 팀원 코딩 컨벤션, 코딩 스타일, 그라운드룰 협의
- 깃허브 세팅(커밋,플로우 등등)

### 2단계

- 프로젝트 설계 및 구현
- 코드리뷰, 코드리뷰 방식에 대한 학습
- 클린코드에 대한 공부
- 페어 프로그래밍(함께 프로그래밍하며 머리는 한 개가 아닌 두 개가 있으면 좋다)
- 라이브 코딩 (오프라인)

### 3단계

- unit 테스트 강화 (테스트 케이스 추가)
- test coverage 측정
- CI/CD 구축 (git action)
- code refactoring

### 4단계

- 성능 측정
- UI/UX 강화
- 프로젝트 발표
- Web 마스터 도구
- 고도화

## 느낀점

1. github project, wiki 와 비슷하다
2. 다른 점이라면 일정 관리, 스프린트 계획 관리에 더 집중이 된 것 같음
3. 저희가 도입, 정리하고자 했던 버킷? 리스트들에 대한 정리가 더욱 명확하게 할 수 있을 것 같음
4. 문서와 버킷리스트의 태스크들을 직접적으로 연결시킬 수 있기에 문서화에도 조금 더 편할 것 같음
5. 위키와 다르게 Confluence 는 노션과 같아서 사용이 더욱 편할 것 같음
6. 다행스럽게도 아직 프로젝트 진행을 하지 않았기에 도입은 더욱 간단할 것 같음 (프로젝트 진행 중에 도입은 빡셈)
